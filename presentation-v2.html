Use Case Selection

  • We chose Use Case 1: Payment Dispute System
  • Internal banking operations tool for staff
  • Deterministic rules engine with full reasoning transparency
  • Three payment types, four routing actions, three priority levels

Where Do We Start?

  • At first, we had no idea how to begin
  • Typical architect muscle flexing said:
  • "Use Kiro to help us start this process"
  • Turned out to be the best decision we made

Generating Requirements

  • Took the use case description → asked Kiro to create initial_requirements.md
  • Manual review confirmed quality — minimal changes needed
  • Kiro generated a great set of requirements out of the box
  • Asked for suggestions on other artifacts:
        Design, UI Spec, API Spec, Architecture

EARS Format Requirements

  • Focused heavily on reworking into EARS format
  • Multiple passes and checks to ensure correctness
  • Requirements file = cornerstone of all reviews
  • Added example template file for future requirements
  • Next step: Include as a steering rule for automatic enforcement

Test Cases

  • Generated all test cases in GIVEN/WHEN/THEN format
  • Reviewed for correctness against EARS requirements
  • Added template file for future test case files
  • Next step: Include as a steering rule for consistent formatting

Reconciling Documentation

  • Reconciled efforts into the conference-required format
  • Applied the same iterative process to all required files:
  •   Requirements → Test Cases → API Spec → Architecture → UI Spec
  • Template-first approach ensured consistency across all docs

Trimming Requirements

  • Judging rules required lowering from 38 to 21 requirements
  • Kiro suggested intelligent merging and removal strategies
  • Lost very little of the original scope
  • Still covered the full dispute triage journey
  • Impressed with the quality of suggestions

Cross-Verification

  • Multiple iterations syncing requirement changes to all documents
  • Cross-verification checks between:
  •   Requirements ↔ Test Cases
  •   Requirements ↔ API Spec
  •   Requirements ↔ UI Spec
  • Ensured full traceability across artifacts

UI Styling

  • Updated ui_spec.md with styles "borrowed" from standardbank.co.za
  • Scraped color palette, typography, spacing, component styles
  • Eventually yielded great results — felt like a real banking app
  • Standard Bank Blue #0033A1 became our signature

Wireframes & Pre-Hook Checks

  • Generated wireframes to visually verify the user journey
  • Great way to build confidence in what we'd created
  • First pre-hook check added:
      Always verify wireframes before generating UI code
  • Future priority: Immediate feedback via hooks at every step

Confidence Check

  • Great tip: ask Kiro its confidence level to generate a working project
  • Scored 92–95% confidence
  • Gave us the green light to proceed with implementation
  • Should be a hook or standard practice for future projects

Building the Solution

  • Submitted docs and specs
  • Started building and playing with the solution
  • Kiro generated the full stack:
  •   Express Prisma + SQLite React + Vite
  • Triage engine with deterministic rules

What We Learned

  • It's our job to understand the concepts surrounding spec-driven development
  • Use Kiro to suggest next steps when stuck — even at the very beginning!
  • The quality of all artifacts is far superior to anything our team could produce, even with 10× the time
  • Documentation and specs really are the most important artifacts in this new world
  • Think harness and auto-verification of specs constantly — we barely scratched the surface
  • Just do it! Practice makes perfect
